# CO core is a pipeline from primitives to gates + variant sweep

CO must be described as a pipeline: primitives (density, entropy, loopiness) feed diagnostics, diagnostics feed gates (collapse, merge, headers), and gates emit actions plus compute budgets. Each part must be specified before tuning, and variant switches (H0-H3 agents) reuse the same knobs across tasks, as AI_17 emphasizes. This is the gap-free framework you asked for: derived bricks (diagnostics → gate) combine into a full CO core, ready to be evaluated via the preregistered grids/seeds.

## Philosophical Justification
- A pipeline makes the system auditable: primitives → diagnostics → gates → actions/budgets. Without explicit staging, tuning is ad hoc.
- Variant switches reuse the same knobs, enabling fair comparison across tasks (aligned with evaluation grids).

## Clarifications / Further Context
- Primitives: density, entropy, loopiness; Diagnostics: measures feeding gate decisions; Gates: collapse, merge, headers.
- Variants H0–H3 reuse knobs; preregistered grids/seeds ensure reproducibility.

## Next Steps in Chain
- suggest: document each gate and its diagnostics; publish the grid configurations.
