# Architecture Doctrine

This file states the architectural doctrine of the CO kernel.

## 1. Core principle

The kernel is not a flat generic optimizer.

It is a change-first structural cognition/control system.

It does not begin from static substance-like objects as primitive givens.

It begins from structured unfolding and derives stabilized regularities from there.

## 2. Major architectural layers

The canonical target-state architecture distinguishes:

1. meta-header
2. header
3. primitives
4. elements
5. element groups
6. final fusion with classical stream
7. translator boundaries

## 3. Meta-header

The meta-header provides the prior regime frame:
- rule stability
- prior classicality
- prior expected staticity/dynamism

## 4. Header

The header provides live runtime regime modulation:
- sensitivity
- reevaluation pressure
- structural cadence
- reopening of stabilized assumptions
- local classicality modulation

## 5. Primitives

Primitives are reusable structural lenses or basis terms.

They are not full mechanisms by themselves.

## 6. Elements

Elements are mechanism families built from primitives, combinators, transforms, and weights.

They emit typed structural contribution packets.

## 7. Groups

Elements may be fused into higher-order groups such as:
- `EA + EB`
- `EA - EB`
- weighted/gated/order-sensitive combinations

## 8. Final fusion

Group outputs must then fuse with:
- header contribution
- meta-header contribution
- explicit classical/STOA continuation stream

The final result is a CO continuation surface.

## 9. Translators

Translators are first-class boundary operators.

They convert:
- task-local input into path-space updates
- continuation surfaces into concrete task actions
- task-local feedback into kernel-relevant updates

## 10. Internal representation

The kernel’s native internal representation is:
- path-based
- time-free
- asymmetry-preserving
- branch-space-aware

It should not be reduced to flat task-state features.

## 11. Asymmetry

Asymmetry is foundational.

It must live in:
- representation
- update semantics
- and some composition/fusion behavior

## 12. Classicality

Classicality is not the enemy of CO.

Classicality is a regime judgment:
- some domains are sufficiently fixed/stabilized that near-total classical collapse is the most honest outcome

CO does not need to fight STOA in such regimes.
